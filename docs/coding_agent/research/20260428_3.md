## 2. 项目定位

本项目是运行在 Linux 环境下的 Python CLI 工具，用于监控单个业务脚本的一次执行。

调度职责由 `crontab` 承担。任务监视器负责启动业务脚本、等待结束或超时、读取结构化日志、判定结果、推送飞书并退出。

第一版不是常驻服务，不是任务调度系统，不做多任务并发调度。

| 缓存文件损坏 | 自动重建 |

### 3.2 `20260428_2` 中确认的 BUG 修正

原结构化日志路径设计为：

```text
${structured_log_root}/${date}/${job}_${number}.log
```

该设计存在问题：业务脚本不知道监视器生成的当前执行编号 `number`，因此无法可靠写出带编号的日志文件。

修正后的结构化日志路径为：

```text
${structured_log_root}/${date}/${job}.log
```

业务脚本每次执行向同一个任务日志文件追加一行 JSON Lines，不覆盖旧内容。

监视器使用自身缓存生成的执行次数 `number` 与结构化日志文件的行数进行对比：

- 如果 `number > 日志条数`：说明本次业务执行没有产生对应结构化日志，判定结构化日志异常，并飞书告知。
- 如果 `number <= 日志条数`：读取最后一条日志，视作本次任务产生的结构化日志。

## 4. 更新后的关键业务约束

### 4.1 单次只监视一个任务

命令行协议固定为：

```bash
task_monitor --config <config_path> --job <job_name>
```

一次进程运行只处理一个任务。

### 4.2 任务由监视器启动

业务脚本由监视器启动。第一版不向业务脚本传递业务参数，不捕获业务脚本 `stdout/stderr`。

`interpreter` 允许包含参数，例如：

```toml
interpreter = "python3 -u"
```

实现时不使用 `shell=True`。建议使用：

```text
shlex.split(interpreter) + [script]
```

业务脚本路径在启动前必须存在且是文件。

### 4.3 结构化日志协议

结构化日志路径规则更新为：

```text
${structured_log_root}/${date}/${job}.log
```

格式为 JSON Lines。业务脚本每次执行追加一行，不覆盖文件。

每行格式：

```json
{"status":"success","message":"finished"}
```

合法 `status`：

- `success`
- `failed`

`message` 缺失时使用空字符串；`message` 非字符串时转为字符串。

监视器读取策略：

1. 统计结构化日志文件中的有效物理行数。
2. 对比本次缓存执行编号 `number`。
3. 如果 `number > 行数`，本次结构化日志缺失。
4. 如果 `number <= 行数`，读取最后一行作为本次结构化日志。

### 4.4 判定优先级

最终状态判定优先级仍为：

```text
timeout -> structured log -> process exit code
```

补充说明：

- 超时发生后直接判定为 `timeout`，不读取后续结构化日志作为最终结果。
- 未超时时，如果 `number > 日志条数`，视为结构化日志异常，再按退出码兜底。
- 如果结构化日志最后一行非法，也视为结构化日志异常，再按退出码兜底。

### 4.5 超时不终止业务进程

超时处理必须满足：

- 捕获 `subprocess.TimeoutExpired`
- 不调用 `kill()`
- 不调用 `terminate()`
- 不继续等待业务进程
- 推送飞书后监视器退出

### 4.6 执行编号先持久化

执行编号必须在业务脚本启动前生成并持久化。

同一时间可能存在多个监视器实例，因此缓存读改写必须使用 `fcntl.flock`。

缓存文件损坏时自动重建。自动重建应记录日志，重建后的当前任务编号从 `1` 开始。

## 5. 模块影响分析

### 5.1 `context.py`

结构化日志路径应从：

```text
${structured_log_root}/${date}/${job}_${number}.log
```

改为：

```text
${structured_log_root}/${date}/${job}.log
```

`RunContext` 仍保留 `number`，用于和日志行数对比。

### 5.2 `structured_log.py`

结构化日志模块不再只表达“读取文件最后一行”，还需要返回日志行数信息。

建议 `StructuredLogResult` 增加或包含：

```text
line_count: int
line_count_sufficient: bool
```

职责变为：

- 文件不存在时返回 `found=False`、`line_count=0`。
- 文件存在时统计行数。
- 当 `number > line_count` 时返回 `valid=False` 并说明本次日志缺失。
- 当 `number <= line_count` 时读取最后一行并解析。

### 5.3 `evaluator.py`

判定矩阵需要纳入“执行次数大于日志行数”的场景。

该场景属于结构化日志异常：

- 如果业务退出码为 `0`：最终状态为 `success_with_log_error`，必须飞书通知。
- 如果业务退出码非 `0` 或启动失败：最终状态为 `failed`，必须飞书通知。

### 5.4 `notifier.py`

飞书内容应包含更新后的结构化日志路径：

```text
${structured_log_root}/${date}/${job}.log
```

建议额外包含：

- 执行编号 `number`
- 结构化日志行数 `line_count`
- 是否满足 `number <= line_count`

便于定位业务脚本未写结构化日志的问题。

## 7. 调研结论

第三版方案应以“按任务单文件追加 JSON Lines”为结构化日志协议。

编码阶段需要特别落实：

1. `RunContext.structured_log_path` 使用 `${structured_log_root}/${date}/${job}.log`。
2. `StructuredLogResult` 记录日志行数和行数是否满足本次执行编号。
3. `structured_log.py` 在 `number > line_count` 时返回结构化日志异常。
4. `evaluator.py` 将行数不足纳入结构化日志异常兜底逻辑。
5. 飞书通知内容包含执行编号、日志行数和结构化日志路径。
6. 保持超时优先，超时后不读取后续日志作为最终结果。
