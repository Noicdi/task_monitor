## 2. 项目定位

本项目是运行在 Linux 环境下的 Python CLI 工具，用于监控单个业务脚本的一次执行。

调度职责由 `crontab` 承担，任务监视器只负责：

1. 加载配置。
2. 初始化监视器日志。
3. 生成并持久化本次执行编号。
4. 启动业务脚本。
5. 等待业务脚本结束或超时。
6. 读取业务脚本生成的结构化日志。
7. 判定最终状态。
8. 按规则推送飞书通知。
9. 返回退出码并退出。

第一版不是常驻服务，不是任务调度系统，不做多任务并发调度。

## 4. 关键业务约束

### 4.1 单次只监视一个任务

命令行协议固定为：

```bash
task_monitor --config <config_path> --job <job_name>
```

一次进程运行只处理一个任务。

### 4.2 任务由监视器启动

业务脚本由监视器启动。第一版不向业务脚本传递业务参数，不捕获业务脚本 `stdout/stderr`。

`interpreter` 已确认允许包含参数，例如：

```toml
interpreter = "python3 -u"
```

实现时不应使用 `shell=True`。建议使用 `shlex.split(interpreter)` 拆分解释器命令，再追加脚本路径：

```text
shlex.split(interpreter) + [script]
```

业务脚本路径在启动前必须存在且是文件。

### 4.3 结构化日志优先

结构化日志路径规则：

```text
${structured_log_root}/${date}/${job}_${number}.log
```

格式为 JSON Lines，监视器只读取最后一行。

合法 `status`：

- `success`
- `failed`

`message` 缺失时使用空字符串；`message` 非字符串时转为字符串。

### 4.4 判定优先级

最终状态判定优先级为：

```text
timeout -> structured log -> process exit code
```

超时一旦发生，本次监视器直接判定为 `timeout`，不再读取后续结构化日志作为最终结果。

### 4.5 超时不终止业务进程

超时处理必须满足：

- 捕获 `subprocess.TimeoutExpired`
- 不调用 `kill()`
- 不调用 `terminate()`
- 不继续等待业务进程
- 推送飞书后监视器退出

### 4.6 执行编号先持久化

执行编号必须在业务脚本启动前生成并持久化。同一时间可能存在多个监视器实例，因此缓存读改写必须使用 `fcntl.flock`。

缓存文件损坏时按用户反馈自动重建。自动重建应记录日志，重建后的当前任务编号从 `1` 开始。

## 5. 配置协议

配置文件使用 TOML，包含：

- `global`
- `feishu`
- `jobs.<job>`

运行期依赖策略：

- Python 3.11+ 使用标准库 `tomllib`
- Python 3.8 ~ 3.10 使用 `tomli`
- HTTP 使用标准库 `urllib.request`
- 子进程使用 `subprocess.Popen`
- 文件锁使用 Linux `fcntl.flock`
- 命令拆分使用标准库 `shlex`

除 `tomli; python_version < '3.11'` 外，不新增运行期依赖。

## 6. 模块边界理解

| 模块 | 职责 |
| --- | --- |
| `cli.py` | 解析命令行参数，调用主流程 |
| `config.py` | 加载和校验 TOML 配置 |
| `context.py` | 构造本次任务运行上下文 |
| `cache.py` | 管理每日任务执行编号 |
| `runner.py` | 启动业务脚本并等待结果 |
| `structured_log.py` | 读取并解析结构化日志最后一行 |
| `evaluator.py` | 根据超时、结构化日志、退出码生成最终结果 |
| `notifier.py` | 构造并发送飞书消息 |
| `monitor_logger.py` | 初始化和记录监视器自身日志 |
| `orchestrator.py` | 编排完整执行流程 |

关键依赖约束：

- `runner.py` 不依赖通知模块。
- `structured_log.py` 不依赖进程执行模块。
- `evaluator.py` 负责统一结果判定。
- `orchestrator.py` 是唯一流程编排中心。
- 飞书 HTTP 调用逻辑只能位于 `notifier.py`。

## 7. 错误处理补充理解

### 7.1 配置错误

如果配置文件无法读取、TOML 非法或无法取得飞书 Webhook，则只写日志或 stderr 并退出。

### 7.2 监视器日志初始化失败

如果 `monitor_log_dir` 无法初始化，允许降级到 stderr 继续记录监视器错误。

### 7.3 通知失败

如果本次结果需要通知但飞书通知失败：

- 保留原业务判定结果用于日志记录。
- 进程退出码追加返回 `7`。

这里的“追加返回 `7`”理解为：最终进程退出码使用 `7` 表示通知失败，同时日志和通知失败记录中保留原始业务状态与原始业务退出码。

### 7.4 缓存损坏

缓存文件损坏时自动重建，不再按缓存错误直接退出。该行为会丢失当日已记录计数，因此必须记录 WARNING 或 ERROR 级别日志。

## 9. 调研结论

第一版实现应按“轻量 CLI + 标准库优先 + 明确模块边界”推进。

本轮用户反馈后，编码阶段需要特别落实：

1. `interpreter` 使用 `shlex.split` 支持参数，但不启用 shell。
2. 业务脚本启动前校验路径存在且是文件。
3. 缓存损坏自动重建并记录日志。
4. `message` 非字符串转字符串。
5. 配置无法读取 Webhook 时只写日志或 stderr 并退出。
6. 监视器日志不可用时降级 stderr。
7. 通知失败时最终退出码返回 `7`，但保留原业务状态记录。
