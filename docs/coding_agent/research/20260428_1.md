## 2. 项目定位

本项目是一个运行在 Linux 环境下的 Python CLI 工具，用于监控单个业务脚本的一次执行。

它不是常驻服务，也不是任务调度系统。调度职责仍由 `crontab` 承担，任务监视器只负责在一次 cron 触发中：

1. 加载配置。
2. 生成本次任务执行编号。
3. 启动业务脚本。
4. 等待业务脚本结束或超时。
5. 读取业务脚本生成的结构化日志。
6. 判定最终状态。
7. 按规则推送飞书通知。
8. 退出。

## 3. 关键业务约束

### 3.1 单次只监视一个任务

命令行协议固定为：

```bash
task_monitor --config <config_path> --job <job_name>
```

一次进程运行只处理一个 `job`，不做多任务调度或并发执行。

### 3.2 任务由监视器启动

业务脚本启动方式固定为：

```text
<interpreter> <script>
```

第一版不向业务脚本传递业务参数，不捕获业务脚本 `stdout/stderr`。

### 3.3 结构化日志优先

业务脚本自行写 JSON Lines 结构化日志，路径规则为：

```text
${structured_log_root}/${date}/${job}_${number}.log
```

监视器只读取最后一行。合法状态只允许：

- `success`
- `failed`

`message` 只用于通知展示，不解析业务语义。

### 3.4 判定优先级

最终状态判定优先级为：

```text
timeout -> structured log -> process exit code
```

这意味着只要发生超时，即使后续业务脚本仍在运行并写出结构化日志，本次监视器也直接判定为 `timeout`。

### 3.5 超时不终止业务进程

超时处理是第一版最重要的边界之一：

- 捕获 `subprocess.TimeoutExpired`
- 不调用 `kill()`
- 不调用 `terminate()`
- 不继续等待业务进程
- 不读取后续结构化日志作为最终结果
- 推送飞书后监视器退出

### 3.6 执行编号必须先持久化

执行编号 `number` 必须在业务脚本启动前生成并持久化，避免监视器异常退出后重复使用同一编号。

同一时间可能存在多个监视器实例，因此缓存读改写必须在 `fcntl.flock` 保护下完成。

## 4. 配置协议

配置文件使用 TOML，包含三个顶层区域：

- `global`
- `feishu`
- `jobs.<job>`

运行期依赖策略：

- Python 3.11+ 使用标准库 `tomllib`
- Python 3.8 ~ 3.10 使用 `tomli`
- HTTP 使用标准库 `urllib.request`
- 子进程使用 `subprocess.Popen`
- 文件锁使用 Linux `fcntl.flock`

除 `tomli; python_version < '3.11'` 外，第一版不应引入新的运行期依赖。

## 5. 模块边界理解

设计文档指定的模块边界如下：

| 模块 | 职责 |
| --- | --- |
| `cli.py` | 解析命令行参数，调用主流程 |
| `config.py` | 加载和校验 TOML 配置 |
| `context.py` | 构造本次任务运行上下文 |
| `cache.py` | 管理每日任务执行编号 |
| `runner.py` | 启动业务脚本并等待结果 |
| `structured_log.py` | 读取并解析结构化日志最后一行 |
| `evaluator.py` | 根据超时、结构化日志、退出码生成最终结果 |
| `notifier.py` | 构造并发送飞书消息 |
| `monitor_logger.py` | 初始化和记录监视器自身日志 |
| `orchestrator.py` | 编排完整执行流程 |

关键依赖约束：

- `runner.py` 不依赖通知模块。
- `structured_log.py` 不依赖进程执行模块。
- `evaluator.py` 负责统一结果判定。
- `orchestrator.py` 是唯一流程编排中心。
- 飞书 HTTP 调用逻辑只能位于 `notifier.py`。

## 7. 错误处理理解

错误分为两类：

1. 业务任务结果：`success`、`success_with_log_error`、`failed`、`timeout`
2. 监视器自身错误：`monitor_error`

退出码需要和最终状态、错误来源对应：

| 退出码 | 含义 |
| ---: | --- |
| `0` | 任务成功，且必要通知成功 |
| `1` | 任务失败 |
| `2` | 任务超时 |
| `3` | 配置错误 |
| `4` | 缓存错误 |
| `5` | 脚本启动错误 |
| `6` | 结构化日志异常但业务退出码为 0 |
| `7` | 飞书通知失败 |
| `10` | 未预期的监视器内部错误 |

## 10. 调研结论

第一版应按“轻量 CLI + 标准库优先 + 明确模块边界”的方式实现。

实现前不应扩展以下能力：

- 业务参数传递
- 任务并发调度
- 自动重试
- Web UI
- 多 Webhook
- 任务级 Webhook 覆盖
- stdout/stderr 捕获
- 超时杀进程
