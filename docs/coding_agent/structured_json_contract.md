# Structured JSON Log Contract

This document describes the structured JSON log that business scripts must
generate for this repository. It is intended as a direct reference for coding
agents that need to create or modify job scripts monitored by `task_monitor`.

## Overview

The required output is a JSON Lines log file, not a single JSON document. Each
business script run must append exactly one JSON object line to the current
day's job log file.

The monitor runs one configured job per invocation, then reads the structured
log file and evaluates only the last line. It also compares the monitor's cached
execution number with the file line count to detect missing output for the
current run.

Evidence:

- `README.md`: structured logs are appended as one JSON object per run.
- `src/task_monitor/structured_log.py`: `read_structured_log()` reads all lines,
  checks the line count, and parses only the last line.
- `src/task_monitor/context.py`: `build_context()` constructs the structured
  log path.

## Path

Business scripts must append to:

```text
${structured_log_root}/${date}/${job}.jsonl
```

Variables:

| Variable | Meaning | Source |
| --- | --- | --- |
| `structured_log_root` | Root directory for business structured logs | Config file `[global].structured_log_root` |
| `date` | Current monitor date in `YYYYMMDD` format | Generated by `orchestrator.run()` |
| `job` | Job name | CLI `--job`, matching `[jobs.<job>]` in config |

Example:

```text
/tmp/task_structured_logs/20260428/daily_sync.jsonl
```

Important: do not generate `${job}_${number}.jsonl`. The current implementation
uses one append-only `${job}.jsonl` file per job per day. The monitor keeps the
execution number internally and uses it only to validate that enough log lines
exist.

## File Format

The file must be UTF-8 text using JSON Lines:

- One complete JSON object per line.
- Append mode is required.
- Each run should append one trailing newline after its JSON object.
- The last non-empty run result must be the last line after the script finishes.

Minimal success example:

```json
{"status":"success","message":"finished"}
```

Failure example:

```json
{"status":"failed","message":"upstream api returned 500"}
```

## Fields

| Field | Required | Type | Valid values / rule | Meaning |
| --- | --- | --- | --- | --- |
| `status` | Yes | string | `success` or `failed` | Business result consumed by the monitor |
| `message` | No | string preferred | Missing becomes `""`; non-string values are converted with `str()` | Human-readable result detail used in evaluation and notification text |

Extra fields are currently ignored by the monitor. They may be present, but
agents should not rely on them affecting monitor behavior.

## Validation Rules

The monitor treats the structured log as invalid when any of the following is
true:

- The path does not exist.
- The path exists but is not a regular file.
- The file cannot be read as UTF-8 text.
- The file is empty.
- The monitor execution number is greater than the file line count.
- The last line is empty.
- The last line is not valid JSON.
- The parsed last line is not a JSON object.
- `status` is missing or is not `success` / `failed`.

The monitor reads the last line with `strip()`, parses it with `json.loads()`,
and requires the parsed payload to be a `dict`.

## Evaluation Impact

If the structured log is valid:

- `status = "success"` produces final status `success`.
- `status = "failed"` produces final status `failed`.
- `message` becomes the evaluation message.

If the structured log is invalid:

- If the business process exit code is `0`, final status becomes
  `success_with_log_error`, and notification is required.
- If the business process exit code is non-zero, final status becomes `failed`,
  and notification is required.

If the business process times out, structured log reading is skipped and the
final status is `timeout`.

## Business Script Requirements

A business script should:

1. Derive or receive the same `structured_log_root` and `job` used by the
   monitor configuration.
2. Use the current date in `YYYYMMDD` format.
3. Create `${structured_log_root}/${date}` if it does not exist.
4. Open `${structured_log_root}/${date}/${job}.jsonl` in append mode.
5. Write one JSON object containing at least `status`.
6. Write a newline.

Python example:

```python
import json
from datetime import datetime
from pathlib import Path

structured_log_root = Path("/tmp/task_structured_logs")
job = "daily_sync"
date = datetime.now().strftime("%Y%m%d")

path = structured_log_root / date / "{}.jsonl".format(job)
path.parent.mkdir(parents=True, exist_ok=True)

record = {
    "status": "success",
    "message": "finished",
}

with path.open("a", encoding="utf-8") as fh:
    fh.write(json.dumps(record, ensure_ascii=False, separators=(",", ":")))
    fh.write("\n")
```

## Internal Cache File

The monitor maintains its own execution counter at:

```text
${cache_dir}/execution_numbers.json
```

Typical shape:

```json
{"date":"20260428","jobs":{"daily_sync":1}}
```

Business scripts and coding agents should not generate or modify this file when
creating business structured logs. It is owned by `src/task_monitor/cache.py`.

## Code Evidence

- `src/task_monitor/config.py`: reads `[global].structured_log_root`,
  `[global].cache_dir`, and job tables from TOML config.
- `src/task_monitor/orchestrator.py`: generates `date` as `YYYYMMDD`, obtains
  the execution number, runs the business script, and then reads the structured
  log unless the process timed out.
- `src/task_monitor/context.py`: builds
  `${structured_log_root}/${date}/${job}.jsonl`.
- `src/task_monitor/structured_log.py`: implements line-count validation,
  JSON parsing, required `status` validation, and `message` handling.
- `src/task_monitor/evaluator.py`: maps structured log results to final monitor
  status and exit code.
- `tests/test_context.py`: asserts the structured log path uses `${job}.jsonl`
  without the execution number.
- `tests/test_structured_log.py`: covers missing files, line shortage, last-line
  parsing, non-string `message`, and invalid `status`.
