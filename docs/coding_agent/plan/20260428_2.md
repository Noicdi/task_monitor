# 任务监视器开发方案

## 1. 本轮目标

根据 `docs/design/1.txt` 和用户对 `20260428_1` 的反馈，形成第二版开发方案。

当前仍处于文档审阅阶段。未获得用户明确批准最终 `plan.md` 前，不编写源码、不添加测试、不修改构建文件。

## 2. 相对上一版的变化

本轮吸收以下用户确认结论：

1. 配置错误且无法加载飞书 Webhook 时，只写日志并退出。
2. `monitor_log_dir` 初始化失败时，允许降级输出到 stderr。
3. 飞书通知失败时，不覆盖原业务结果；最终进程退出码返回 `7`，日志保留原业务状态。
4. 结构化日志 `message` 非字符串时转字符串。
5. `interpreter` 允许包含参数。
6. 业务脚本路径启动前必须存在且是文件。
7. 缓存文件损坏时自动重建。

## 3. 实现原则

1. 严格按第一版范围实现，不扩展非目标能力。
2. 不新增未批准的运行期依赖。
3. 保持模块职责单一。
4. 所有状态判定集中在 `evaluator.py`。
5. 所有流程编排集中在 `orchestrator.py`。
6. 飞书 HTTP 调用只位于 `notifier.py`。
7. 业务脚本超时后不终止业务进程。
8. 支持 `interpreter` 参数，但不使用 `shell=True`。

## 5. 模块设计

### 5.1 `cli.py`

职责：

- 使用 `argparse` 解析 `--config` 和 `--job`。
- 调用 `orchestrator.run(config_path, job_name)`。
- 将返回的退出码传给 `sys.exit()`。

### 5.2 `config.py`

职责：

- 加载 TOML 配置。
- 校验必填字段和字段类型。
- 构造配置数据结构。
- Python 3.11+ 使用 `tomllib`，Python 3.8 ~ 3.10 使用 `tomli`。

配置加载失败且无法取得 Webhook 时，不尝试飞书通知，只通过日志或 stderr 输出后退出。

### 5.3 `context.py`

职责：

- 根据任务名、日期、编号和配置构造 `RunContext`。
- 计算结构化日志路径。

### 5.4 `cache.py`

职责：

- 在 `${cache_dir}/execution_numbers.json` 维护执行编号。
- 使用 `fcntl.flock` 保护读取、修改、写回。
- 日期变化后重置计数。
- 缓存文件损坏时自动重建，并记录日志。

写回建议使用临时文件加 `replace`，避免半写入。

### 5.5 `runner.py`

职责：

- 校验业务脚本路径存在且是文件。
- 使用 `shlex.split(interpreter) + [script]` 生成命令参数。
- 使用 `subprocess.Popen(args)` 启动业务脚本。
- 使用 `process.wait(timeout=timeout_seconds)` 等待。
- 返回 `ProcessResult`。

禁止：

- 禁止使用 `shell=True`。
- 超时后禁止调用 `kill()` 或 `terminate()`。
- 禁止捕获业务脚本 `stdout/stderr`。

### 5.6 `structured_log.py`

职责：

- 读取结构化日志最后一行。
- 解析 JSON object。
- 校验 `status`。
- 提取 `message`。

处理规则：

- 文件不存在、空文件、JSON 非法、`status` 非法：`valid=False`。
- `message` 缺失：使用空字符串。
- `message` 非字符串：使用 `str(message)`。

### 5.7 `evaluator.py`

职责：

- 实现判定优先级：

```text
timeout -> structured log -> process exit code
```

- 生成 `EvaluationResult`。
- 决定是否需要通知。
- 保留原始业务状态，供通知失败时记录。

### 5.8 `notifier.py`

职责：

- 构造飞书 text 消息。
- 使用 `urllib.request` 发送 POST JSON。
- 返回通知成功或失败。

约束：

- 只支持全局 Webhook。
- 不支持任务级 Webhook 覆盖。
- 不引入第三方 HTTP 客户端。

### 5.9 `monitor_logger.py`

职责：

- 初始化 `${monitor_log_dir}/${date}.log`。
- 初始化失败时降级 stderr。
- 记录关键执行过程。

### 5.10 `orchestrator.py`

职责：

- 编排完整流程。
- 捕获可预期异常并映射退出码。
- 通知失败时返回退出码 `7`，但日志保留原始业务最终状态。

建议流程：

```text
load config
init logger or stderr fallback
generate execution number
build run context
run script
if timed out:
    evaluate timeout
else:
    read structured log
    evaluate result
if should_notify:
    send feishu notification
    if notification failed:
        log original result and return 7
return mapped exit code
```

## 6. 退出码策略

基础退出码仍遵循设计文档：

| 退出码 | 含义 |
| ---: | --- |
| `0` | 任务成功，且必要通知成功 |
| `1` | 任务失败 |
| `2` | 任务超时 |
| `3` | 配置错误 |
| `4` | 缓存错误 |
| `5` | 脚本启动错误 |
| `6` | 结构化日志异常但业务退出码为 0 |
| `7` | 飞书通知失败 |
| `10` | 未预期的监视器内部错误 |

用户确认后的补充规则：

- 如果需要通知且通知失败，进程最终返回 `7`。
- 原始业务状态、业务退出码、结构化日志状态仍写入监视器日志。
- 缓存文件损坏自动重建，不直接返回 `4`；但缓存目录不可写、锁失败、写入失败仍可返回 `4`。

## 7. 测试方案

### 7.1 配置测试

- 正常配置加载。
- 缺失顶层字段。
- 缺失任务字段。
- 任务名不存在。
- `timeout_seconds` 非法。
- `notify_on_success` 非布尔值。

### 7.2 缓存测试

- 同日同任务递增。
- 同日不同任务分别计数。
- 日期变化后重置。
- 缓存损坏自动重建。
- 缓存目录不可写或写入失败返回缓存错误。

### 7.3 执行器测试

- 单命令解释器。
- 带参数解释器，例如 `python3 -u`。
- 脚本不存在。
- 脚本路径不是文件。
- 正常退出码。
- 非零退出码。
- 启动失败。
- 超时不终止业务进程。

### 7.4 结构化日志测试

- 文件不存在。
- 空文件。
- 单行合法 JSON。
- 多行只取最后一行。
- JSON 非法。
- `status` 缺失。
- `status` 非 `success/failed`。
- `message` 缺失。
- `message` 非字符串时转字符串。

### 7.5 判定测试

- 超时优先。
- 合法 `success`。
- 合法 `failed`。
- 日志异常且退出码为 0。
- 日志异常且退出码非 0。
- 启动失败。
- 通知失败返回 `7` 且保留原状态记录。

### 7.6 编排测试

- 成功不通知。
- 成功按配置通知。
- 失败通知。
- 超时通知。
- 结构化日志异常通知。
- 配置不可读且无法获取 Webhook 时只写 stderr 或日志。
- 日志目录不可用时降级 stderr。

## 8. 不纳入第一版的内容

- 常驻服务。
- 多任务并发调度。
- 任务自动重试。
- Web UI。
- 数据库存储。
- 捕获业务脚本 stdout/stderr。
- 解析普通业务日志。
- 向业务脚本传递业务参数。
- 超时后终止业务进程。
- 多飞书机器人。
- 任务级 Webhook 覆盖。
- 复杂权限系统。

## 9. 审阅点

请重点审阅：

1. `interpreter` 使用 `shlex.split` 支持参数是否满足预期。
2. 缓存损坏自动重建后编号从 `1` 开始是否可接受。
3. 通知失败时“返回 `7`，但日志保留原始业务结果”的解释是否符合“追加返回 7”的含义。

用户明确批准本 `plan` 后，才进入编码阶段。
