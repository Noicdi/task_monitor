# 任务监视器开发方案

## 1. 本轮目标

根据 `docs/design/1.txt` 实现第一版 Linux Python 任务监视器。

当前阶段只形成开发方案，等待用户审阅和批准。未获批准前不进入编码阶段。

## 2. 实现原则

1. 严格按设计文档实现第一版能力。
2. 不引入未批准的新依赖。
3. 不实现设计文档列入非目标的功能。
4. 保持模块职责单一，避免跨模块反向依赖。
5. 运行期优先使用 Python 标准库。
6. 所有状态判定集中在 `evaluator.py`。
7. 所有流程编排集中在 `orchestrator.py`。
8. 飞书 HTTP 发送只放在 `notifier.py`。

## 4. 模块设计

### 4.1 `cli.py`

职责：

- 使用 `argparse` 解析 `--config` 和 `--job`。
- 调用 `orchestrator.run(config_path, job_name)`。
- 将返回的退出码传给 `sys.exit()`。

不负责业务判定、配置细节或通知发送。

### 4.2 `config.py`

职责：

- 加载 TOML 配置。
- 校验必填字段。
- 构造 `GlobalConfig`、`FeishuConfig`、`JobConfig`。
- 根据 Python 版本选择 `tomllib` 或 `tomli`。

约束：

- 不发送飞书。
- 不启动业务脚本。
- 不创建执行上下文。

### 4.3 `context.py`

职责：

- 根据配置、任务名、日期和执行编号构造 `RunContext`。
- 计算结构化日志路径。

约束：

- 不负责编号生成。
- 不读取结构化日志。

### 4.4 `cache.py`

职责：

- 在 `global.cache_dir` 下维护 JSON 缓存。
- 使用 `fcntl.flock` 保护读改写。
- 按日期和任务名生成单调递增编号。
- 在业务脚本启动前持久化编号。

建议缓存文件：

```text
${cache_dir}/execution_numbers.json
```

关键实现点：

- 确保缓存目录存在。
- 加锁后读取、校验、更新、写回。
- 日期变化时重置 `jobs` 计数。
- 写回建议使用临时文件加 `replace`，降低半写入风险。

### 4.5 `runner.py`

职责：

- 使用 `subprocess.Popen([interpreter, script])` 启动业务脚本。
- 使用 `process.wait(timeout=timeout_seconds)` 等待。
- 返回 `ProcessResult`。

超时约束：

- 捕获 `TimeoutExpired`。
- 不调用 `kill()`。
- 不调用 `terminate()`。
- 返回 `timed_out=True`。

### 4.6 `structured_log.py`

职责：

- 读取结构化日志最后一行。
- 解析 JSON object。
- 校验 `status`。
- 提取 `message`。
- 返回 `StructuredLogResult`。

边界：

- 文件不存在、空文件、JSON 非法、字段非法均返回 `valid=False`。
- `message` 缺失时按空字符串处理。

### 4.7 `evaluator.py`

职责：

- 根据 `ProcessResult`、`StructuredLogResult` 和 `JobConfig.notify_on_success` 生成 `EvaluationResult`。
- 实现统一判定优先级：

```text
timeout -> structured log -> process exit code
```

通知规则：

- `success` 根据 `notify_on_success` 决定。
- `success_with_log_error` 必须通知。
- `failed` 必须通知。
- `timeout` 必须通知。
- `monitor_error` 必须通知。

### 4.8 `notifier.py`

职责：

- 构造飞书 text 消息。
- 使用 `urllib.request` 发送 POST 请求。
- 返回通知成功或失败结果。

约束：

- 只使用全局 Webhook。
- 不支持任务级 Webhook。
- 不引入第三方 HTTP 客户端。

### 4.9 `monitor_logger.py`

职责：

- 初始化标准库 `logging`。
- 日志文件路径为 `${monitor_log_dir}/${date}.log`。
- 记录监视器关键运行过程。

### 4.10 `orchestrator.py`

职责：

- 串联完整流程。
- 捕获可预期错误并映射退出码。
- 确保失败、超时、监视器错误按规则通知。

建议流程：

```text
load config
validate job
init monitor logger
generate execution number
build run context
run script
if timed out:
    evaluate timeout
else:
    read structured log
    evaluate result
notify if needed
return exit code
```

## 5. 测试方案

### 5.1 配置测试

覆盖：

- 正常配置加载。
- 缺失顶层字段。
- 缺失任务字段。
- 任务名不存在。
- `timeout_seconds` 非法。
- `notify_on_success` 非布尔值。

### 5.2 缓存测试

覆盖：

- 同日同任务递增。
- 同日不同任务分别计数。
- 日期变化后重置。
- 缓存损坏时报错。

### 5.3 结构化日志测试

覆盖：

- 文件不存在。
- 空文件。
- 单行合法 JSON。
- 多行只取最后一行。
- JSON 非法。
- `status` 缺失。
- `status` 非 `success/failed`。
- `message` 缺失。

### 5.4 判定测试

覆盖设计文档中的结果判定矩阵：

- 超时优先。
- 合法 success。
- 合法 failed。
- 日志异常且退出码为 0。
- 日志异常且退出码非 0。
- 启动失败。

### 5.5 执行器测试

覆盖：

- 正常退出码。
- 非零退出码。
- 启动失败。
- 超时不终止业务进程。

### 5.6 编排测试

覆盖：

- 成功不通知。
- 成功按配置通知。
- 失败通知。
- 超时通知。
- 结构化日志异常通知。
- 通知失败返回 `7`。

## 6. 风险与处理

### 6.1 配置加载前无法获取 Webhook

如果配置文件不存在或 TOML 解析失败，监视器可能无法取得飞书 Webhook。

建议策略：

- 记录到 stderr。
- 如果配置已部分加载且能取得 Webhook，再尝试通知。
- 编码前建议用户确认是否需要环境变量兜底。

### 6.2 `interpreter` 是否允许参数

设计文档同时描述“解释器路径或命令”和固定启动方式 `<interpreter> <script>`。

建议第一版保守处理为单个可执行命令，不解析 shell 字符串，不使用 `shell=True`。

如果需要支持 `python3 -u`，应在设计上将 `interpreter` 改为数组或新增字段，避免 shell 注入风险。

### 6.3 飞书失败退出码覆盖

设计文档规定飞书推送失败返回非零退出码，退出码表给出 `7`。

建议只要“应通知但通知失败”，最终进程退出码返回 `7`，同时在日志中保留原始业务状态。

### 6.4 监视器日志初始化失败

如果 `monitor_log_dir` 不可写，属于监视器自身错误。

建议返回 `monitor_error` 对应退出码，但是否还能推送取决于配置是否已加载成功。

## 7. 不纳入第一版的内容

以下能力不在本轮实现：

- 常驻服务。
- 多任务并发调度。
- 任务自动重试。
- Web UI。
- 数据库存储。
- 捕获业务脚本 stdout/stderr。
- 解析普通业务日志。
- 向业务脚本传参。
- 超时后终止业务进程。
- 多飞书机器人。
- 任务级 Webhook 覆盖。
- 复杂权限系统。

## 8. 需要用户审阅确认

请重点确认：

1. 是否接受第一版 `interpreter` 只支持单个命令，不支持内嵌参数。
2. 配置加载失败且无法读取 Webhook 时，是否允许只写 stderr 并退出。
3. 通知失败是否统一覆盖最终退出码为 `7`。
4. 缓存文件损坏是否直接报错，不自动重建。
5. `message` 非字符串时是否按结构化日志非法处理。

用户批准本方案后，才进入编码阶段。
