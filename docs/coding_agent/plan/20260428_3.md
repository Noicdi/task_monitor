## 2. 相对上一版的变化

本轮核心变化是修正结构化日志协议。

上一版：

```text
${structured_log_root}/${date}/${job}_${number}.log
```

第三版：

```text
${structured_log_root}/${date}/${job}.log
```

业务脚本每次执行追加一行 JSON Lines。监视器使用自身缓存的执行编号 `number` 与日志行数对比：

- `number > line_count`：本次结构化日志缺失，判定结构化日志异常并飞书通知。
- `number <= line_count`：读取最后一行作为本次结构化日志。

## 3. 实现原则

1. 严格按第一版范围实现，不扩展非目标能力。
2. 不新增未批准的运行期依赖。
3. 保持模块职责单一。
4. 所有状态判定集中在 `evaluator.py`。
5. 所有流程编排集中在 `orchestrator.py`。
6. 飞书 HTTP 调用只位于 `notifier.py`。
7. 业务脚本超时后不终止业务进程。
8. 支持 `interpreter` 参数，但不使用 `shell=True`。
9. 结构化日志按任务单文件追加，不按执行编号分文件。

## 5. 数据结构调整

### 5.1 `RunContext`

```text
RunContext
  job_name: str
  date: str
  number: int
  structured_log_path: Path
  config_path: Path
  job_config: JobConfig
  global_config: GlobalConfig
```

`structured_log_path` 使用：

```text
${structured_log_root}/${date}/${job}.log
```

`number` 继续保留，用于校验日志行数。

### 5.2 `StructuredLogResult`

建议结构：

```text
StructuredLogResult
  found: bool
  valid: bool
  status: Optional[str]
  message: Optional[str]
  error_message: Optional[str]
  raw_line: Optional[str]
  line_count: int
  expected_number: int
  line_count_sufficient: bool
```

说明：

- `line_count` 表示结构化日志文件当前物理行数。
- `expected_number` 表示本次缓存执行编号。
- `line_count_sufficient` 表示 `line_count >= expected_number`。

## 6. 模块设计

### 6.1 `cli.py`

职责：

- 使用 `argparse` 解析 `--config` 和 `--job`。
- 调用 `orchestrator.run(config_path, job_name)`。
- 将返回的退出码传给 `sys.exit()`。

### 6.2 `config.py`

职责：

- 加载 TOML 配置。
- 校验必填字段和字段类型。
- 构造配置数据结构。
- Python 3.11+ 使用 `tomllib`，Python 3.8 ~ 3.10 使用 `tomli`。

配置加载失败且无法取得 Webhook 时，不尝试飞书通知，只通过日志或 stderr 输出后退出。

### 6.3 `context.py`

职责：

- 根据任务名、日期、编号和配置构造 `RunContext`。
- 计算结构化日志路径。

路径规则：

```text
${structured_log_root}/${date}/${job}.log
```

不再生成 `${job}_${number}.log`。

### 6.4 `cache.py`

职责：

- 在 `${cache_dir}/execution_numbers.json` 维护执行编号。
- 使用 `fcntl.flock` 保护读取、修改、写回。
- 日期变化后重置计数。
- 缓存文件损坏时自动重建，并记录日志。

执行编号必须在业务脚本启动前持久化。

### 6.5 `runner.py`

职责：

- 校验业务脚本路径存在且是文件。
- 使用 `shlex.split(interpreter) + [script]` 生成命令参数。
- 使用 `subprocess.Popen(args)` 启动业务脚本。
- 使用 `process.wait(timeout=timeout_seconds)` 等待。
- 返回 `ProcessResult`。

禁止：

- 禁止使用 `shell=True`。
- 超时后禁止调用 `kill()` 或 `terminate()`。
- 禁止捕获业务脚本 `stdout/stderr`。

### 6.6 `structured_log.py`

职责：

- 根据 `RunContext.structured_log_path` 读取 `${job}.log`。
- 统计日志文件物理行数。
- 使用本次执行编号 `number` 校验日志行数。
- 在 `number <= line_count` 时读取最后一行。
- 解析 JSON object。
- 校验 `status`。
- 提取 `message`。

处理规则：

- 文件不存在：`found=False`、`line_count=0`、`valid=False`。
- 空文件：`found=True`、`line_count=0`、`valid=False`。
- `number > line_count`：`line_count_sufficient=False`、`valid=False`。
- `number <= line_count`：解析最后一行。
- JSON 非法或 `status` 非法：`valid=False`。
- `message` 缺失：使用空字符串。
- `message` 非字符串：使用 `str(message)`。

### 6.7 `evaluator.py`

职责：

- 实现判定优先级：

```text
timeout -> structured log -> process exit code
```

- 生成 `EvaluationResult`。
- 决定是否需要通知。
- 将“日志行数不足”视为结构化日志异常。
- 保留原始业务状态，供通知失败时记录。

结构化日志异常包括：

- 文件不存在。
- 空文件。
- `number > line_count`。
- 最后一行 JSON 非法。
- `status` 缺失或非法。

兜底规则：

- 结构化日志异常且退出码为 `0`：`success_with_log_error`，必须通知。
- 结构化日志异常且退出码非 `0`：`failed`，必须通知。

### 6.8 `notifier.py`

职责：

- 构造飞书 text 消息。
- 使用 `urllib.request` 发送 POST JSON。
- 返回通知成功或失败。

通知内容应包含：

- 任务名
- 日期
- 执行编号
- 最终状态
- 业务 message
- 业务脚本退出码
- 是否超时
- 结构化日志路径
- 结构化日志行数
- 日志行数是否满足本次执行编号
- 业务脚本路径
- 主机名
- 发生时间

### 6.9 `monitor_logger.py`

职责：

- 初始化 `${monitor_log_dir}/${date}.log`。
- 初始化失败时降级 stderr。
- 记录关键执行过程。

日志中应包含结构化日志路径、行数、执行编号和解析结果。

### 6.10 `orchestrator.py`

职责：

- 编排完整流程。
- 捕获可预期异常并映射退出码。
- 通知失败时返回退出码 `7`，但日志保留原始业务最终状态。

建议流程：

```text
load config
init logger or stderr fallback
generate execution number and persist it
build run context with ${job}.log path
run script
if timed out:
    evaluate timeout
else:
    read structured log with expected number
    evaluate result
if should_notify:
    send feishu notification
    if notification failed:
        log original result and return 7
return mapped exit code
```

## 7. 退出码策略

| 退出码 | 含义 |
| ---: | --- |
| `0` | 任务成功，且必要通知成功 |
| `1` | 任务失败 |
| `2` | 任务超时 |
| `3` | 配置错误 |
| `4` | 缓存错误 |
| `5` | 脚本启动错误 |
| `6` | 结构化日志异常但业务退出码为 0 |
| `7` | 飞书通知失败 |
| `10` | 未预期的监视器内部错误 |

补充规则：

- 如果需要通知且通知失败，进程最终返回 `7`。
- 原始业务状态、业务退出码、结构化日志状态仍写入监视器日志。
- 缓存文件损坏自动重建，不直接返回 `4`。
- 缓存目录不可写、锁失败、写入失败仍可返回 `4`。

## 8. 测试方案

### 8.1 配置测试

- 正常配置加载。
- 缺失顶层字段。
- 缺失任务字段。
- 任务名不存在。
- `timeout_seconds` 非法。
- `notify_on_success` 非布尔值。

### 8.2 缓存测试

- 同日同任务递增。
- 同日不同任务分别计数。
- 日期变化后重置。
- 缓存损坏自动重建。
- 缓存目录不可写或写入失败返回缓存错误。

### 8.3 上下文测试

- 结构化日志路径为 `${structured_log_root}/${date}/${job}.log`。
- 路径不包含执行编号。
- `RunContext.number` 正确保留。

### 8.4 执行器测试

- 单命令解释器。
- 带参数解释器，例如 `python3 -u`。
- 脚本不存在。
- 脚本路径不是文件。
- 正常退出码。
- 非零退出码。
- 启动失败。
- 超时不终止业务进程。

### 8.5 结构化日志测试

- 文件不存在。
- 空文件。
- 单行合法 JSON 且 `number=1`。
- 多行合法 JSON 且读取最后一行。
- `number > line_count` 时返回结构化日志异常。
- `number <= line_count` 时读取最后一行。
- JSON 非法。
- `status` 缺失。
- `status` 非 `success/failed`。
- `message` 缺失。
- `message` 非字符串时转字符串。

### 8.6 判定测试

- 超时优先。
- 合法 `success`。
- 合法 `failed`。
- 日志行数不足且退出码为 `0`，得到 `success_with_log_error`。
- 日志行数不足且退出码非 `0`，得到 `failed`。
- JSON 非法且退出码为 `0`，得到 `success_with_log_error`。
- JSON 非法且退出码非 `0`，得到 `failed`。
- 启动失败。
- 通知失败返回 `7` 且保留原状态记录。

### 8.7 编排测试

- 成功不通知。
- 成功按配置通知。
- 失败通知。
- 超时通知。
- 日志行数不足通知。
- 结构化日志最后一行非法通知。
- 配置不可读且无法获取 Webhook 时只写 stderr 或日志。
- 日志目录不可用时降级 stderr。
